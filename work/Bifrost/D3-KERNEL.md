## §1 拍板结论

**D3 = A 方案·轮廓局部化**：恒星主体走纯材质假体积，仅在边缘轮廓 / 日珥弧处叠 dynamic mesh（复用 NS_Slime marching cubes）做真融合翻涌立体感。

**理由**：
- **性能可控**：整球 dynamic mesh（B/C 方案的极端）在近景主机位下屏占大 × 采样密，成本失控；A 方案让主体走"便宜的假体积"、只在最需要立体感的轮廓处付真 mesh 成本。
- **breakdown 价值保住**：marching cubes 融合表面仍是可拆解、可 flex 的技术点，只是从"整球"降为"轮廓"。
- **契合美术形态**：等离子恒星的视觉签名主要在边缘（日珥、翻涌的融合"舌"），主体表面允许假体积，与 D2 冷神性的雾化边缘处理天然吻合。

**否决 B/C 的原因**：
- **B 纯 shader 假体积**：完全没有真 mesh，"marching cubes 融合"这个 breakdown 卖点消失，Part 1 前沿内核叙事站不住。
- **C 转真体积（GaussianVolume）**：R&D 最厚（换栈+驱动天气链路重接），且内核身份从 isosurface 漂移到体积渲染，等于砸 D1 定案。作品集时间盒下不合算。

## §2 A 方案落地：材质三层堆

（与 `PIPELINE.md` §2 完全一致，此处只做 A 方案身份的技术拆解）

| 层 | 内容 | 成本 | 是否 A 方案独占 |
|---|---|---|---|
| **层 1 · 本体表面（主力）** | 大球 mesh + 自定义材质：多层 3D 噪声（domain warp）+ 时间演化；emissive 拉出冷白过曝核心 + icy cyan 边缘梯度 | 低 | 是（B 方案也用，A/B 共享底） |
| **层 2 · 边缘 isosurface 细节（A 方案核心）** | 复用 NS_Slime marching cubes，仅在**边缘轮廓 / 日珥弧**处叠真 dynamic mesh，做融合翻涌立体感 | 中（局部而非整球） | **A 方案独占** |
| **层 3 · 光源（因果链物理落点）** | 恒星驱动方向光 + 大范围光 + 大气散射，真实照亮整个海岸 | 低 | 是（三方案共通） |

**层 2 是 A 方案的技术身份**：既保 marching cubes 的 breakdown 价值，又不让整球吃 mesh 成本。

## §2.5 层 2 触发判据（拍板：路线 C）

> **拍板结论（2026-07-09）**：选**路线 C · screen-space 粗筛 + world-space 场二次筛选**。理由见本节末尾，详细实现步骤待细化。

**问题**：A 方案拍板了"仅轮廓/日珥弧触发局部 mesh"，但**"该在哪触发"从未落地过判据**——ROADMAP §4 依赖图写"⟸ D3 已定"直接跳过了这一步，§4 性能预算（2-4ms）目前是空头承诺，全靠这个判据的实现开销撑不撑得住。

两条技术路线，特征完全不同：

| 维度 | 路线 A · Screen-space silhouette | 路线 B · World-space 场驱动 mask |
|---|---|---|
| **判据本体** | 法线·视线夹角接近 90°（`abs(dot(N, V)) < threshold`） | 复用层 1 的域扭曲噪声场，噪声值超阈值区域=该翻涌，与"日珥弧"物理直觉一致 |
| **能捕捉什么** | 纯几何轮廓带（球体边缘一圈） | 表面下的翻涌形态，能长出真正的"日珥弧"，不受限于几何轮廓 |
| **实现复杂度** | 低：per-vertex/per-pixel 算 N·V，阈值转 mesh 触发区 | 中高：需要噪声场从材质侧"读出来"喂给 marching cubes 的密度场输入，两套系统（材质 Shader + Niagara/CS marching cubes）要共享同一份场数据 |
| **额外 GPU 开销** | 几乎 0（复用现成法线/视线，无额外 pass） | 需要至少一次额外 compute pass 提取/采样噪声场到 marching cubes 可读的 volume texture，开销未知，直接影响 §4 的 2-4ms 预算是否能兑现 |
| **与层 1 材质耦合** | 弱耦合：轮廓判据和材质纹路是两套独立逻辑，容易出现"轮廓 mesh 位置和材质翻涌纹路对不上"的贴皮感 | 强耦合：同一份噪声场驱动材质纹路和 mesh 触发区，视觉上天然一致，但要求两个系统间传数据的管线打通 |
| **走位测试风险**（ROADMAP §5 提醒：不能只测 S3 主机位） | 低：轮廓带随视角自动跟随摄像机，任何角度都稳定成立 | 中：如果场是静态烘焙的，玩家从不同角度看同一时刻的"该翻涌区域"可能显得固定不变、缺乏随视角变化的鲜活感；需要确认噪声场本身带时间演化 |
| **breakdown 叙事** | 平："轮廓检测"是常见技术，flex 点较弱 | 强："磁场/噪声驱动的真实翻涌区域，非贴皮轮廓"更能撑起 Part 1 前沿内核叙事 |

**路线 C（已选定）**：screen-space 轮廓带 **作为触发区域的粗筛**（先框定"哪一圈算轮廓"），再用世界空间噪声场在这个粗筛范围内做二次筛选决定"这一圈里具体哪几处该长出日珥弧"——把路线 B 的额外开销限制在已经被路线 A 缩小的小范围内，是开销和视觉效果的平衡点，代价是引入两套判据串联的调试复杂度。

**选 C 否 A/B 的理由**：
- 否 A（纯屏幕空间轮廓）：轮廓判据和层 1 材质纹路两套独立逻辑，容易"轮廓 mesh 位置和翻涌纹路对不上"的贴皮感——这会直接削弱 D3 作为"前沿内核"的 breakdown 叙事，等于自己拆选 A 方案（否 B）时立的台。
- 否 B（纯世界空间场驱动）：额外 compute pass 开销未知，对已经绷紧的 5 周路线 C 工时（`ROADMAP.md` §2）是隐患，且整球范围内做噪声场提取比局部粗筛后再做贵得多。
- 选 C：用路线 A 的低开销粗筛把路线 B 的强叙事效果（磁场驱动真实翻涌）限制在小范围，性能可控的同时不丢叙事强度。

**这个判据直接决定的下游影响**：
- §4 性能预算 2-4ms 是否现实（路线 C 需要先做一次实测才能填数）
- breakdown 素材里"整球 vs 轮廓局部化的 overdraw 热图"对比截图（§5）该怎么标注触发依据
- 层 1 材质的域扭曲噪声参数（domain warp 强度/时间演化速度）需要为路线 C 预留"可被外部读取"的接口

**前置确认结果（2026-07-09 · 用户已在 UE 里查证，已完成）**：`NS_Slime` 密度场输入接受**外部 3D SDF（有符号距离场）**，不是任意噪声值/volume texture。这比预期更规整（SDF 天然适配 marching cubes/isosurface），但也带来一个新的管线要求：层 1 材质的域扭曲噪声是标量场，**不能直接喂给 `NS_Slime`**，需要先转换成 SDF 表示（例如噪声阈值化后生成距离场，或用现成 SDF 生成工具/CS 处理一遍）再接入。世界空间二次筛选那一步的具体实现，要把"噪声场 → SDF"这层转换算进管线设计和性能预算里。

## §3 UE5.8 集成路径

- **NS_Slime 现有实现**：已在个人 UE study 工程 PreComputedGodRay 之外的模块中跑通 marching cubes / metaball 融合表面。可复用面：融合逻辑核 + Niagara Emitter 结构；已知坑：整球模式下 overdraw 直线拉高（正是走轮廓局部化的动因）。
- **"驱动天气"衍生效果的连接**：恒星光源（层 3）直接驱动方向光 + Volumetric Fog + 体积云（`PIPELINE.md` §3 raymarching 框架），**不需要额外自定义 Shading Model**。
- **UE5.8 自定义渲染的接入点**：光源走标准 Directional Light + Sky Atmosphere + Volumetric Cloud（自定义 raymarch 走 Post Process Material 或自定义 volume shader）。恒星本体材质走 Emissive/Unlit + 自定义 depth。层 2 的 marching cubes mesh 走 Niagara Mesh Renderer 或直接 Static Mesh。
- **MCP/Agent 介入边界**：L3 内核**全部人手调**（材质节点图 · 时间演化 · 冷神性色彩梯度 · 层 2 的 emitter 参数）；MCP 只负责 L1 量产（`PIPELINE.md` §5）。

## §4 成本 / 性能校准

**基调**：性能是"可管理性 note"，唯一底线"必须实时"（≥30fps 在个人办公机 RTX 3080 上）。价值排序 = breakdown 价值 / 技术亮点 > 画面说服力 > 性能。

**A 方案实测预期**（W2 恒星本体成型后回填实测数据）：
- 层 1 纯材质：预算 <1ms（与普通天空盒同量级）
- 层 2 轮廓 marching cubes：预算 2-4ms（局部化后可控，vs 整球方案 8-15ms）
- 层 3 光源 + 大气：预算 1-2ms
- 合计目标 ≤ 8ms（留出体积云 5-8ms + 场景其他 15ms 的余量）
