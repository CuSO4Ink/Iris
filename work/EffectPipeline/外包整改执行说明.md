# BYDC 外包特效整改执行说明

> 面向：漫行者外包团队  
> 依据：`notes/外包交付规范.md`、`notes/外包归档清单.md`  
> 目标：把现有 BYDC 外包特效整改为可 **Content 级迁移导入主工程**（自研 UE 5.7.3）并稳定播放、对齐 Sequence 的完整交付包。

## 0. 一句话结论

这次整改不是"补几个文件"，而是要求每个镜头重新整理成一个**完整、可独立打开、依赖闭包完整、路径命名规范、插件声明准确**的 UE 工程交付包。引擎版本一般沿用即可，无需为版本号专门升级重存；**唯 C30 须改用公版引擎交付**（当前为 GUID 自定义引擎关联，未验证），**ZibraVDB 镜头须交 `.zibravdb` 源**（见 §1.2 与 §2）。

交付后，主工程侧应能做到：

1. 打开工程不报缺失引用。
2. 关卡内特效 Actor 正确绑定资产。
3. Sequence / Level 中特效能正常播放。
4. ZibraVDB / HoudiniNiagara 等插件资产依赖完整。
5. 资产可按统一目录导入主工程，不污染现有 Content。

## 1. 全部镜头必须统一整改的通用要求

### 1.1 工程完整性

每个镜头必须交付完整 UE 工程，不接受裸 Content、散装资产或未解压压缩包。

标准结构：

```text
<ShotName>_UE57/
  <ShotName>.uproject
  Config/
  Content/
    FX_<ShotName>/
      Level/
      Niagara/
      PointCache/
      Geo/
      Volume/
      Material/
      Texture/
      Mesh/
```

必须包含：

| 必含项 | 要求 |
|---|---|
| `.uproject` | 位于工程根目录 |
| `Config/` | 至少包含正常打开工程所需配置 |
| `Content/` | 所有资产统一放在 `Content/FX_<镜头号>/` 下 |
| 预览说明 | 标明主关卡、主 Sequence、入口资产 |

禁止：

- 只交付 `Content/`。
- 只交付 `.7z`、`.zip` 未解压包。
- 工程嵌套在 `UE/Cloud/` 这类二级目录里。
- 资产散落在 `houdini/`、`abc/`、`zibra/`、`WorkProjects/Outsource/` 等历史目录中。

### 1.2 引擎版本

引擎版本**一般不强制升级**，只要资产能被主工程正常 **migrate / 导入**、且不依赖外包方本机私有引擎路径即可。两点例外：

- **C30 须改用公版引擎交付**：其当前为 GUID 自定义引擎关联，尚未验证能否被标准引擎打开；用公版 UE 5.5 交付即可。
- **ZibraVDB 镜头须交付 `.zibravdb` 源**：`ZibraVDBAssetData` 是预编译私有压缩格式，跨版本/跨引擎可能"资产在、体积解不出"，须由我方用源在自研引擎侧重导（详见 §2）。

### 1.3 插件声明

`.uproject` 的 `Plugins` 只允许启用实际使用的插件。

| 插件 | 何时启用 |
|---|---|
| `ZibraVDB` | 镜头确实使用 ZibraVDB Volume |
| `HoudiniEngine` | 镜头使用 Houdini 烘焙 / HDA 流程 |
| `HoudiniNiagara` | 镜头使用 HoudiniPointCache 驱动 Niagara |
| `NiagaraFluids` | 镜头确实使用 Niagara Fluids |
| `MovieRenderPipeline` | 镜头交付了可用渲染队列或 Sequence 验证流程 |

必须删除与特效无关的 VP / 虚拟制片 / DMX / nDisplay / VirtualScouting 等插件声明。

特别注意：

- 文件名含 `vdb` 不等于 ZibraVDB。
- UE 原生 `SparseVolumeTexture` / `VolumeTexture` 不需要按 ZibraVDB 处理。
- C60 当前疑似就是“启用了 ZibraVDB，但资产实际是 UE 原生 SVT”的错误案例。

### 1.4 Content 目录与命名

每个镜头统一整理为：

```text
Content/FX_<镜头号>/
  Level/        关卡 .umap
  Niagara/      Niagara System / Emitter
  PointCache/   HoudiniPointCache
  Geo/          GeometryCache / Alembic
  Volume/       ZibraVDB uasset + .zibravdb 源文件
  Material/     Material / MI
  Texture/      Texture / SVT
  Mesh/         Static Mesh / Skeletal Mesh
```

命名要求：

| 类型 | 命名示例 |
|---|---|
| 关卡 | `<镜头号>_gk.umap`，例如 `B20_gk.umap` |
| Niagara System | `NS_<镜头号>_<描述>` |
| HoudiniPointCache | `HPC_<镜头号>_<描述>` |
| GeometryCache | `GC_<镜头号>_<描述>` |
| ZibraVDB 资产 | `VDB_<镜头号>_<描述>` |
| 材质 | `M_<镜头号>_<描述>` / `MI_<镜头号>_<描述>` |
| 贴图 | `T_<镜头号>_<描述>` |

禁止：

- `Main.umap`、`NewMap.umap` 这类通用名。
- `NewNiagaraSystem2`、`CurveA1` 这类无业务含义名字。
- C60 资产里残留 C30 命名。
- D20 的 `/Game/Asstes/` 拼写错误路径。
- `StarterContent/`、`Developers/`、`EditorMoviePipelineQueue.uasset` 等无关内容。

### 1.5 引用与依赖闭包

交付前必须在外包方工程内完成：

1. 打开主关卡。
2. 修复所有 Missing Reference / Missing Object。
3. 执行 Fix Up Redirectors。
4. 保存关卡和相关资产。
5. 重新打开工程验证无缺失引用。

所有内部引用必须指向本工程 `Content/FX_<镜头号>/` 内部资产，不允许引用外包方本地磁盘路径、旧工程路径、其他镜头路径或拼写错误路径。

如果一个镜头依赖另一个镜头资产，有两种合格做法：

| 做法 | 要求 |
|---|---|
| 拆除跨镜头依赖 | 把引用改成本镜头自己的资产 |
| 纳入依赖闭包 | 把被依赖资产一并交付，并在说明中写明依赖关系 |

不允许交付“打开后再去别的镜头目录找资产”的工程。

## 2. ZibraVDB 镜头专项要求

如果镜头使用 ZibraVDB，必须同时交付三层资产：

| 层级 | 必须交付 |
|---|---|
| 源数据 | `.zibravdb` 源文件 |
| UE 资产 | 从源文件导入的 `ZibraVDBAssetData` uasset |
| 关卡引用 | 关卡内 ZibraVDBActor 正确引用上述资产 |

整改要求：

1. `.zibravdb` 源文件放入 `Content/FX_<镜头号>/Volume/Source/` 或同级清晰目录。
2. `ZibraVDBAssetData` uasset 放入 `Content/FX_<镜头号>/Volume/`。
3. 关卡内 ZibraVDBActor 必须绑定有效 Volume，不允许 Volume 为 None。
4. 若现有 uasset 内含压缩数据，也必须补交独立 `.zibravdb` 源文件，方便主工程侧重建与验证。
5. 若镜头实际不使用 ZibraVDB，应删除 ZibraVDB 插件声明和无效 Actor。

当前已知：

| 镜头 | 当前状态 | 外包必须处理 |
|---|---|---|
| D20 | 关卡内有 36 处 ZibraVDB 引用，但无 Volume uasset、无 `.zibravdb` 源 | 必须补交对应 Volume uasset 和 `.zibravdb` 源；或确认这些 ZibraVDB Actor 已废弃并从关卡中删除 |
| A45 | 有 6 个 ZibraVDBAssetData uasset，平级 `UE_to_vdb/A45/output/vol/` 已带 6 个 `.zibravdb` 源 | 源已在但在工程目录外，需随工程交付并归入 `Volume/Source/`，与 uasset 一一对应 |
| B20 | 有 9 个 uasset + 9 个 `.zibravdb` 源，闭包最完整 | 保留源文件，但需按新目录整理并清理无关内容 |
| C50 | 有 6 个 ZibraVDBAssetData uasset，平级 `vdb/zibra/` 已带 7 个 `.zibravdb` 源，但源体积极小（约 280~410KB） | 确认是正式效果资产还是低模/测试资产；若正式，随工程交付源；若测试，替换为正式资产 |
| C60 | 资产头部为 UE 原生 SparseVolumeTexture/VolumeTexture | 不要按 ZibraVDB 交付；删除冗余 ZibraVDB 插件声明 |

## 3. HoudiniNiagara 镜头专项要求

如果镜头使用 HoudiniPointCache → Niagara 流程，必须同时交付：

| 必含项 | 要求 |
|---|---|
| HoudiniPointCache | 放入 `PointCache/`，命名为 `HPC_<镜头号>_<描述>` |
| Niagara System | 放入 `Niagara/`，命名为 `NS_<镜头号>_<描述>` |
| Niagara Actor | 关卡中正确引用 Niagara System |
| 缓存说明 | 如果使用 Sim Cache，说明帧数、帧率、起止帧 |

当前 D20 已知有 `NH_baiyin_D20_1/2` 和 `N_baiyin_D20_1/2/3`，但资产平铺且命名不统一，需要整理到标准目录并确认引用不断。
