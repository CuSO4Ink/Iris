# Progressive Disclosure — 渐进式披露准则

> **L1 维护者规则**。修改任何 AI 可见文档时自查。

## 核心原则

**按需暴露**：每份文档只讲当前层该讲的事，深层细节让 AI 按路径主动拉。

## 五条硬约束

1. **单一真相源**：同一条规则只在一处正文叙述，其他地方用路径引用，不复述
2. **入口从瘦到胖**：Onboarding 模板最短（指路），规则文件最详（定义），按调用深度递增
3. **层级隔离**：L1 维护者内容不进 L0，L2 项目内容不进 L1；AI 不该读的就别塞给它
4. **用词极简**：能表格不段落，能路径不描述，删到再删就伤义为止
5. **唯一例外**：canary 规则的硬核一句（"回复末尾带 `唔呣。`"）可在 Onboarding 模板内联——因为这是"不读任何文件也要遵守"的底线

## 自查清单（改文档前过一遍）

- [ ] 我写的这段话，别处有没有同义叙述？如有 → 只保留一处，其他改成引用路径
- [ ] 这份文档的读者真的需要这段吗？如不 → 删
- [ ] 能用表格 / 列表 / 路径替代的段落 → 替代
- [ ] 标题、解释性段落、"为什么这样设计"的叙述 → 除非对执行规则有用，否则搬到 daily memory

## 反模式示例

**反模式**：Onboarding 模板内联"PARA 八大目录地图"——AI 读根 README 就有了。
**正确**：Onboarding 模板只说"结构见根 `README.md`"。

**反模式**：两份文档都列"维护者职责范围"。
**正确**：只在 `rules/maintainer/README.md` 列一次，其他地方引用。
