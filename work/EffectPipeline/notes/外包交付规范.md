# 外包特效交付规范（BYDC）

> 版本：v1.1 | 日期：2026-07-07
> 适用：漫行者外包 BYDC 白银副本特效交付
> 交付方式：**Content 级迁移导入主工程**（主工程为自研 UE 5.7.3），外包工程引擎版本一般不强制升级
> ⚠️ 例外：**C30 须用公版引擎交付**（当前为 GUID 自定义引擎关联，未验证）；**ZibraVDB 镜头须交 `.zibravdb` 源文件**（详见 §5.2）

---

## 一、工程完整性要求

每个特效镜头**必须**交付完整 UE 工程，包含以下三部分：

| 必含项 | 说明 | 当前反面案例 |
|--------|------|-------------|
| `.uproject` 文件 | 工程入口，位于根目录 | E20、C80 缺失 |
| `Config/` 目录 | 含 DefaultEngine.ini 等 | C30、C60、D20、E20、C80 缺失 |
| `Content/` 目录 | 实际资产 | E40 仅 .7z 未解压 |

**禁止交付裸 Content 片段或未解压压缩包。**

## 二、引擎版本与插件

### 2.1 引擎版本
- 引擎版本**一般不强制升级**，只要资产能正常 migrate / 导入主工程即可
- **C30 须用公版引擎交付**：其当前为 GUID 自定义引擎关联，尚未验证能否被标准引擎打开；用公版 UE 5.5 交付即可
- ⚠️ **ZibraVDB 镜头须交付 `.zibravdb` 源文件**：`ZibraVDBAssetData` 为预编译私有压缩格式，跨版本/跨引擎可能"资产在、体积解不出"，须由我方用源在自研引擎侧重导（详见 §5.2）

### 2.2 插件声明
`.uproject` 的 `Plugins` 列表必须**按实际使用情况声明**：

| 插件 | 何时必须启用 |
|------|-------------|
| ZibraVDB | 用了 ZibraVDB Volume 的镜头 |
| HoudiniEngine | 用了 Houdini 烘焙资产的镜头 |
| HoudiniNiagara | 用了 HoudiniPointCache → Niagara 的镜头 |
| NiagaraFluids | 用了 Niagara Fluids 的镜头 |

- **禁止启用与特效无关的插件**（如 VirtualScouting、nDisplay、DMXProtocol 等 VP 插件）
- 资产类型必须和插件声明一致：UE 原生 `SparseVolumeTexture/VolumeTexture` 不等于 ZibraVDB，不应为了文件名含 `vdb` 就启用 ZibraVDB
- 反面案例：A45/B20/C50 启用了 20+ 个 VP 插件，与特效无关；C60 启用 ZibraVDB 但体积资产实为 UE 原生 SVT

## 三、Content 目录结构规范

### 3.1 标准目录结构

```
Content/
  └── FX_<镜头号>/              ← 如 FX_C30/
      ├── Level/                关卡 .umap
      ├── Niagara/              Niagara System / Emitter
      ├── PointCache/           HoudiniPointCache（如有）
      ├── Geo/                  GeometryCache / Alembic（如有）
      ├── Volume/               ZibraVDB 资产 + .zibravdb 源（如有）
      ├── Material/             材质 + MI
      ├── Texture/              贴图
      └── Mesh/                 模型（如有）
```

### 3.2 禁止的目录结构

| 禁止项 | 原因 | 反面案例 |
|--------|------|---------|
| 外包方内部工作流目录名 | 主工程无法解析 | C80 的 `WorkProjects/Outsource/VFX_Project/` |
| 资产平铺无分层 | 无法批量处理、命名冲突 | D20 全部资产堆在 `baiying_D20/` |
| 残留其他镜头命名 | 混淆归属 | C60 的 houdini 目录里全是 C30 命名 |
| UE 模板内容 | 无用冗余 | A45/B20 的 `StarterContent/` |
| 工程嵌套子目录 | 路径混乱 | C50 嵌套在 `UE/Cloud/Content/` |
| Developer 个人目录 | 绑定个人路径 | 多个工程的 `Developers/meow/` |
| 未说明的 `__ExternalActors__` / `__ExternalObjects__` | World Partition 外部 Actor/对象必须与 umap 成套交付并保持路径；若无 WP 需求，禁止交付 | C30/C60 含此目录且存在跨镜头依赖 |
> ⚠️ **WP 豁免（v1.1 新增）**：若工程使用 World Partition 作为场景设置，`__ExternalActors__` / `__ExternalObjects__` 为系统自动生成、Actor 全部保存在此目录下，属正常 WP 产物，允许保留。但须保持路径结构完整，不得缺失或随意重组。（2026-07-07 用户确认 C30/C60 适用此豁免）
| `EditorMoviePipelineQueue.uasset` | 渲染队列临时文件 | C30/C60 含此文件 |

## 四、命名规范

### 4.1 目录与资产命名

- 前缀统一：`<镜头号>_<描述>`，如 `C30_cloud_main`、`D20_ash_particles`
- **禁止拼写错误**——反面案例：D20 内部引用路径 `/Game/Asstes/`（应为 `Assets`）
- **禁止直接复制其他镜头文件不改名**——反面案例：C60 大量 C30 命名残留

### 4.2 关卡命名

- 格式：`<镜头号>_gk.umap`（如 `A45_gk.umap`、`B20_gk.umap`）
- 禁止使用 `Main.umap`、`NewMap.umap` 等通用名

## 五、资产依赖完整性

### 5.1 引用路径

- 所有内部引用路径必须指向 **工程自身 Content 内**，不得引用外包方本地路径
- 交付前必须执行 **Fix Up Redirectors**，确保无悬空 Redirector
- 反面案例：D20 的 `E40_text3/4/5` 是 Redirector，指向拼写错误的 `/Game/Asstes/` 路径

### 5.2 ZibraVDB 体积资产

如果镜头包含 ZibraVDB Volume 特效，必须同时交付：

| 必含项 | 说明 |
|--------|------|
| `.zibravdb` 源文件 | 放在 `Volume/` 目录下，**这是硬性要求**——`ZibraVDBAssetData` uasset 跨版本/跨引擎可能解不出体积，我方需用源在自研 5.7.3 侧重导 |
| ZibraVDBAssetData uasset | 从源文件导入的 UE 资产（随工程交付，但不能只交它） |
| 关卡内 ZibraVDBActor | 正确引用上述资产 |

- ⚠️ 判据：ZibraVDB 镜头不看"资产能否打开"，看"**能否在主工程正常解码显示体积**"；因此源文件是第一交付物
- 反面案例：D20 有关卡引用 36 处 ZibraVDB，但资产和源文件全缺（无源=无法重建，死锁）
- 反面案例：A45/C50 有 `ZibraVDBAssetData` uasset，`.zibravdb` 源虽存在但散落在工程外平级目录（A45 在 `UE_to_vdb/A45/output/vol/`、C50 在 `vdb/zibra/`），未随工程 `Content/` 交付，极易漏交；后续正式交付必须将源随工程一并归入 `Volume/` 目录
- ⚠️ **通用缺源问题（v1.1 新增）**：外包备注说明指出所有 ZibraVDB 镜头均只将 uasset 放入 `Assets/` 子目录、`.zibravdb` 源未随工程交付，此为通用问题（非仅 D20 个案），所有含 ZibraVDB 的镜头均须核查并补交源文件
- ⚠️ **E20 误判修正（v1.1 新增）**：E20 实际使用 ZibraVDB（关卡含 ZebraVDB Actor + Sequence 含 ZebraVDB Playback 轨道），此前误判为仅使用 SparseVolumeMaterial；E20 也须按 ZibraVDB 镜头补交 `.zibravdb` 源

#### 5.2.1 ZibraVDB 空 Actor 豁免（v1.1 新增）

若外包侧因 ZibraVDB 内存占用过大导致工程无法正常打开/保存（如 E40），允许交付**不挂接 ZibraVDB、仅保留空 ZibraVDBActor**的版本。但须满足：

| 要求 | 说明 |
|------|------|
| 空 Actor 保留 | 关卡内保留 ZibraVDBActor（Volume 为空/None），用于主工程侧挂接 |
| `.zibravdb` 源仍须交付 | 即使 Actor 未挂接，源文件必须随工程 `Volume/Source/` 交付，主工程侧自行导入并挂接 |
| Sequence playback 轨道 | 主工程侧挂接后在 Sequence 中添加 ZibraVDB Playback 轨道（外包侧因未挂接故无此轨道，属正常） |
| DELIVERY_NOTE 标注 | 须在交付说明中注明"ZibraVDB 未挂接，主工程侧自行挂接" |

（2026-07-07 用户确认 E40 适用此豁免）

#### 5.2.2 ZibraVDB 前后景遮挡问题（v1.1 新增）

外包备注指出 ZibraVDB 会出现后景遮挡前景粒子的情况。主工程侧导入后需检查：若粒子显示异常，调整 ZibraVDB 的 **Volume Translucency Sort Priority**（如设为 -100）一般可解决前后显示问题。

### 5.3 HoudiniNiagara 资产

如果镜头包含 Houdini → Niagara 流程，必须同时交付：

| 必含项 | 说明 |
|--------|------|
| HoudiniPointCache 资产 | Niagara 的数据源 |
| Niagara System 资产 | 引用上述 PointCache |
| 关卡内 NiagaraActor | 正确引用上述 System |

## 六、交付前自检清单

外包方交付前须逐项确认：

- [ ] 资产可正常 migrate / 导入主工程（引擎版本一般不强制；C30 须用公版引擎交付）
- [ ] `Config/` 目录存在
- [ ] 插件列表仅含实际使用的插件
- [ ] Content 目录遵循 `FX_<镜头号>/` 结构，按类型分子目录
- [ ] 无 `StarterContent/`、`Developers/` 等无关目录；若使用 World Partition，`__ExternalActors__` / `__ExternalObjects__` 为系统自动生成可保留，但须成套交付并保持路径结构完整
- [ ] 无 `Main.umap` / `NewMap.umap` / `EditorMoviePipelineQueue.uasset`
- [ ] 所有资产命名以镜头号开头，无其他镜头命名残留
- [ ] 引用路径无拼写错误，已执行 Fix Up Redirectors
- [ ] ZibraVDB 镜头：`.zibravdb` 源 + Volume uasset 均已交付（⚠️ 所有镜头通用，非仅个别镜头）
- [ ] ZibraVDB 空 Actor 版本（如适用）：源文件已交付，DELIVERY_NOTE 已标注"主工程侧自行挂接"
- [ ] HoudiniNiagara 镜头：PointCache + Niagara System 均已交付
- [ ] 工程可独立打开，无缺失引用警告（ZibraVDB 空 Actor 版本除外）
